Rasa in the Browser: The Same Runtime, A New Host

How Rasa now runs in browser pages through the ras.wasm component, generated WIT adapter assets, and explicit browser capability providers.

A programming language becomes easier to trust when its examples are executable. Not screenshots of results. Not a parallel teaching interpreter. Executable source, running through the same runtime contract that the rest of the product uses.

That is the browser story Rasa now has a concrete path for. A Rasa page can show code, let a reader edit it, run it in place, inspect the result, and still preserve the same language boundary rules used by native and Wasm-hosted execution.

The browser is not a toy host. It has its own capabilities: DOM state, fetch, storage, workers, WebGPU, WebNN, TensorFlow.js, browser permission prompts, and a security model that refuses ambient local power by design. If Rasa is serious about portable semantics and explicit host boundaries, the browser is not a special exception. It is one of the best tests of the design.

This post is about the current browser runtime shape: ras.wasm, WIT session exports, generated component adapter assets, a shared @rasa/browser/engine facade, and capability calls routed through browser-owned providers.

The Product Shape

There are three runtime product names worth knowing:

ras       native Rasa executable
ras-wasmtime  native Wasmtime launcher for the Rasa Wasm runtime product
ras.wasm  Rasa runtime Wasm component

The browser consumes ras.wasm. That matters.

Rasa does not ask the website to maintain a JavaScript implementation of the language. The browser page loads a Wasm component produced from the Rasa runtime product and talks to it through a typed component interface. The JavaScript code around it is a host and UI adapter, not the semantic owner of the language.

The published website carries this shape:

assets/
  ras.wasm
  rasa-component/
    generated browser adapter assets

ras.wasm is the Rasa runtime component. The rasa-component/ assets are generated browser adapter assets derived from it. The exact files in that directory are produced by the component tooling and are not Rasa language concepts. They are not a second runtime and not an alternate semantic path.

Local section builds keep the same shape:

web/model-architecture-explorer/dist/
  ras.wasm
  rasa-component/

That lets each page run standalone during development while the published site can share one runtime asset across sections.

The Runtime Contract

The browser boundary is WIT-shaped:

package rasa:runtime@0.1.0;

interface host {
  call: func(request: list<u8>) -> result<list<u8>, string>;
}

interface session {
  status: func() -> s32;
  render-hash: func() -> u32;
  eval-source: func(source: list<u8>, phase-flags: u32) -> string;
  eval-package-source: func(source: list<u8>, package-bytes: list<u8>, phase-flags: u32) -> string;
  session-new: func() -> u32;
  session-free: func(session: u32);
  session-load-package: func(session: u32, package-bytes: list<u8>) -> string;
  session-eval: func(session: u32, source: list<u8>, phase-flags: u32) -> string;
  inspect-source: func(source: list<u8>, options: u32) -> string;
  syntax-tokens: func(source: list<u8>) -> string;
}

world runtime {
  import host;
  export session;
}

This is a compact but important line in the sand.

The runtime exports session operations: create a session, load packages, evaluate source, inspect source, and ask for syntax tokens. The host provides one import: host.call, the admitted capability boundary.

That split matches Rasa's larger philosophy. Language execution belongs to the runtime. Host power enters only through explicit capability calls.

Build-Time Generation

Browsers do not directly import a Wasm component the way a Wasmtime host does. The website needs JavaScript adapter assets around the component. Rasa generates those assets at build time.

The flow is:

target/rasa-products/ras.wasm
  -> jco component transpilation
  -> target/rasa-web/component-adapters/<hash>/
  -> section dist/rasa-component/
  -> published assets/rasa-component/

The hash cache is deliberate. If the runtime component has not changed, the adapter assets do not need to be regenerated. If ras.wasm changes, the adapter cache key changes with it.

The important boundary is still simple:

ras.wasm owns runtime behavior.
Generated adapter assets own browser component loading.
Browser UI owns presentation and host resources.

That separation keeps the website useful without letting website code become language authority.

The Browser Session Facade

Browser sections use one shared package:

@rasa/browser/engine

It is the browser facade over the generated component adapter. A page asks for a runtime engine, creates a session, and evaluates source through that session:

const engine = await RasaWasmEngine.load("./ras.wasm", { host });
const session = engine.createSession();

await engine.evaluateSessionAsync(session, "(def x 41)", { phases: "all" });
await engine.evaluateSessionAsync(session, "(+ x 1)", { phases: "all" });

The second evaluation sees the x defined by the first one. This is not a stateless request-response evaluator hidden behind a button. It is a stateful Rasa session inside the browser host.

The same facade powers syntax highlighting:

const tokens = engine.syntaxTokens("(+ 1 2 3)");

That choice is small but telling. The editor does not need to invent a separate tokenizer to approximate Rasa. It can ask the runtime-facing facade for the same kind of syntax-token payload used by the rest of the browser experience.

What A Reader Can Run

The smallest proof is ordinary Rasa:

(+ 1 2 3)

Then state:

(def x 41)

And then reuse it:

(+ x 1)

That sequence proves the browser has a real session. It is the shape expected from a Lisp-family REPL: evaluate, define, continue.

The model explorer uses the same runtime path for a more interesting workload. Its Rasa source talks to a browser provider for TensorFlow.js model data:

(ns demo
  (:import [demo.tfjs/model :as model]))

(def m
  (model/load-model :tiny-cnn))

(model/summary m)

The result is not a screenshot baked into the page. The browser session loads a package, evaluates Rasa source, crosses an admitted host boundary, and receives Rasa data back from a browser provider that owns the TensorFlow.js model handle.

That distinction is the point. The page is interactive because Rasa is running, not because the UI simulates a result.

Host Calls Stay Explicit

The browser host bridge is intentionally narrow.

Rasa source
  -> package import
  -> analyzer and IR host-call evidence
  -> admitted capability
  -> WIT host.call import
  -> browser capability provider
  -> encoded Rasa data response

For a tiny test package, the source can look like this:

(ns demo
  (:import [demo.host/test :as host]))

(host/echo 41)

The browser provider receives a canonical request shape:

{:package demo.host/test
 :export echo
 :args [41]}

And it returns Rasa data:

{:status :ok
 :value 42}

The runtime report can include admitted-host-call traces. That gives the page a way to explain what happened: source evaluation, host boundary, provider dispatch, returned value.

This is not just a security mechanism. It is a usability mechanism. When a program crosses into browser power, users should be able to see where and why.

Browser Suspension Without Source Async

The browser host.call WIT shape remains ordinary:

call: func(request: list<u8>) -> result<list<u8>, string>;

The generated browser component adapter lowers that import through JSPI for the stateful session exports. A browser provider can return a JavaScript promise, the shared session facade awaits it, and Rasa source still looks ordinary:

(http/request {:method :get
              :url "https://example.com/data.json"})

That is still intentionally narrow. It is a browser/session boundary, not a new source-visible async API. Rasa's direction remains colorless async: source should not pay syntax tax just because an admitted host operation may suspend. Future WASI 0.3 and component async work can deepen this boundary. The important part is the ownership: suspension belongs to the capability and runtime contract, not to ad hoc user code conventions.

What The Browser Page Can Show

Because the runtime is real, the website can become more than documentation.

A page can show:

  • source code being evaluated;
  • syntax tokens coming from the runtime facade;
  • phase reports for reader, analysis, IR, facts, checks, and eval;
  • admitted host-call traces;
  • provider request and response data;
  • model summaries, graphs, and tables created from live browser resources;
  • reset behavior that creates a fresh runtime session.

That is a different kind of documentation surface. It does not ask a reader to believe that a language has clean boundaries. It lets them run code and inspect the boundary.

Why This Matters

Rasa wants Clojure-family ergonomics without inheriting an ambient-host runtime model. That means the browser cannot be treated as a lower-fidelity demo lane. It has to be a real host with explicit powers, visible limits, and the same semantic runtime contract.

The current browser runtime work proves the first useful slice:

same ras.wasm runtime product
generated component adapter assets
one shared browser session facade
stateful REPL behavior
runtime syntax tokens
admitted host-call bridge
browser provider dispatch
live model explorer workload

That gives Rasa a credible path toward executable documentation, browser-hosted conformance exploration, model inspection, and capability-driven applications that can be read and run in the same place.

The deeper compiler work still matters. WasmGC representation, richer capabilities, colorless async, optimizer proofs, and native-tier execution are not replaced by this. But the browser now has a real place in the product architecture. It can apply pressure on the language from a user-facing surface, not just from a roadmap diagram.

For a language built around inspectable execution, that is the right kind of pressure.
