Milestone 1 planned

CLI Text Transformer

Run a Rasa script that reads text, transforms it, and writes output.

strings, regex, files, stdio, args, exit status

Capability Pressure

  • strings with documented indexing and character semantics
  • regex dialect decision and result model
  • file or stdio input
  • file or stdio output
  • CLI arguments or a small execution context
  • structured runtime errors for missing or denied capabilities

Story Path

  • Practical input/output boundary
    • Choose the smallest non-fixture boundary: CLI arguments, stdin/stdout, file paths, or a host-admitted text buffer.
    • Prove admitted, denied, and unsupported-lane outcomes.
    • Keep host facts out of pure language semantics.
  • Text semantics review
    • Decide the first string indexing and character contract.
    • Name what is portable now and what remains deferred.
  • Regex alpha contract
    • Choose the first regex dialect boundary.
    • Define match, find, replace, split, and capture result shapes.
  • CLI execution context
    • Define how a script sees arguments, stdin, stdout, stderr, and exit status.
    • Keep host facts out of pure language semantics.
  • File/stdio capability alpha
    • Admit read/write operations through explicit capability evidence.
    • Prove denial and missing-capability diagnostics.
  • Demo script and report
    • Add one polished text-transformer example.
    • Show native proof.
    • Show runtime-Wasm proof over the same source-level contract. If blocked, the story cannot close; it must name the implementation blocker and next command.
    • Show a compact capability report rather than requiring users to read raw phase dumps.

Milestone 3 planned

Browser Embedded Rasa

Embed a Rasa evaluator in a browser page with controlled host callbacks.

Wasm session stability, sandboxed capabilities, UI diagnostics

Capability Pressure

  • stable browser Wasm session lifecycle
  • reusable evaluator instance with persistent defs
  • explicit session reset
  • source loading for `rasa.core` through the same mechanism as native where possible
  • structured diagnostics and rendered values for UI consumption
  • host callback admission model for browser-provided functions
  • syntax highlighting or phase output that can support an editor surface

Story Path

  • Browser session contract review
    • Define what a browser evaluator session owns.
    • Confirm how persistent defs, reset, core loading, and diagnostics line up with native runtime sessions.
  • Host callback capability alpha
    • Admit a tiny browser host capability surface.
    • Prove denied or missing callbacks report structured errors.
    • Keep callback identity and JavaScript values outside source semantics.
  • UI result contract
    • Define the result object the page receives after eval.
    • Include rendered value, structured diagnostics, phase status, and enough source span data for UI highlighting.
  • Embedded demo page
    • Build a small page with one Rasa input, one host-provided capability, and a reset button.
    • Reuse shared REPL/editor components where possible.
  • Native parity check
    • Show the same Rasa source can run through a native host harness with an equivalent admitted capability.
    • Record any browser-only or native-only limitations explicitly.
Milestone 4 planned

HTTP JSON Client Script

Fetch JSON, transform data, and emit JSON from Rasa.

JSON, bytes/text boundaries, HTTP client capability, colorless suspension

Capability Pressure

  • JSON parse/generate over Rasa maps, vectors, strings, numbers, booleans, and nil
  • bytes/text boundary decisions for request and response bodies
  • URI or URL validation policy
  • HTTP client capability admission
  • request/response map shape
  • structured errors for denied, unavailable, malformed, timeout, and non-2xx cases
  • suspension evidence without source-level async coloring

Story Path

  • JSON value contract
    • Define JSON-to-Rasa and Rasa-to-JSON mapping.
    • Name numeric, keyword/string key, ordering, and unsupported-value behavior.
  • Body boundary contract
    • Decide when bodies are strings, bytes, or explicit decoded values.
    • Keep encoding failures structured.
  • HTTP request/response data shape
    • Define the minimal request map and response map.
    • Include status, headers, body, and error shape.
  • HTTP client capability alpha
    • Admit one outbound request operation through capability evidence.
    • Prove denied, missing, and unavailable host behavior.
  • Colorless suspension proof
    • Show source code does not need host-specific async syntax.
    • Record where the runtime may suspend and resume.
  • Demo script and report
    • Add a small script that fetches JSON from a controlled test host or fixture.
    • Show native proof.
    • Show browser/Wasm proof through fetch or an admitted host stub, or name the precise deferred host-lowering piece.
Milestone 5 planned

HTTP JSON Service

Serve a small JSON endpoint from Rasa.

routing, request/response maps, lifecycle, concurrency, cancellation

Capability Pressure

  • request map and response map contract
  • JSON body parse/generate reuse from the client milestone
  • header representation and normalization policy
  • HTTP server capability admission
  • route or handler invocation model
  • structured errors for handler failure, malformed response, denied server capability, port binding failure, and unsupported host lanes
  • suspension and cancellation evidence
  • resource cleanup for server lifecycle

Story Path

  • Server request/response contract
    • Reuse or refine the client request/response map shape.
    • Define what an inbound request contains at alpha scope.
  • Handler invocation contract
    • Decide whether handlers are ordinary functions, protocol methods, or a capability-specific adapter shape.
    • Keep handler invocation as Rasa callable semantics, not host callback leakage.
  • Server capability alpha
    • Admit one HTTP server operation through capability evidence.
    • Prove denied, unavailable, malformed response, and handler failure paths.
  • Suspension, cancellation, and lifecycle proof
    • Name where the runtime may park and resume work.
    • Prove server shutdown or cleanup behavior at alpha scope.
    • Record deferred cancellation behavior explicitly.
  • Demo service and report
    • Add one small JSON endpoint demo.
    • Show native proof.
    • Show Wasm proof when a host can provide an inbound request loop, or name the precise host-lowering limitation.
Milestone 6 planned

Capability Model And Manifest Tooling

Generate and validate capability models from provider/interface sources with small human policy overlays.

package loading, capability declarations, importer tooling, generated artifacts

Capability Pressure

  • canonical Rasa capability model
  • importer path from at least one external interface source
  • policy overlay model separated from imported interface facts
  • Rasa-readable declaration surface or restricted data form
  • source-spanned validation diagnostics
  • generated capability catalog artifacts
  • generated denied, missing, and unsupported capability tests
  • documentation/reference projection from the same model

Story Path

  • Capability model boundary
    • Define the Rasa-owned capability model independently of any one manifest syntax.
    • Separate imported interface facts from policy overlay facts.
  • Importer spike
    • Choose one source, likely WIT or existing provider metadata.
    • Generate a small capability model with source/provenance evidence.
  • Policy overlay sketch
    • Add a minimal human-authored overlay for exposure, names, authority, and deferrals.
    • Keep syntax explicitly experimental.
  • Rasa validation diagnostics
    • Validate the generated model and overlay through Rasa tooling.
    • Produce source-spanned diagnostics and suggested fixes for missing policy, unsupported shapes, or unavailable lanes.
  • Artifact generation
    • Generate capability catalog entries, reference docs, test skeletons, and website-capability projection data from the model.
  • Demo package/provider
    • Prove the flow on one tiny provider or package.
    • Show what was generated, what the user wrote, and what Rasa refused.
